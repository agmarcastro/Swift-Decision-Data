## Summary

InfoAgent is a three-layer multi-agent intelligence system that allows C-Level executives at a Brazilian technology retail company to ask natural language questions about their data warehouse and receive real-time, KPI-grounded answers through a Chainlit executive chat interface. The system routes each question through a LangGraph classifier to one of three execution paths: pure SQL generation (Type 1), KPI-formula-grounded SQL using Qdrant-retrieved definitions (Type 2), or hybrid SQL + RAG qualitative enrichment (Type 3). A 100K-row synthetic Star Schema powered by ShadowTraffic provides realistic demo data, and a double-layer READ-ONLY enforcement (PostgreSQL user + MCP regex guard) ensures no agent can modify the data warehouse.

## What Was Built

### Components

| Component | Description |
|-----------|-------------|
| Pydantic Star Schema models | 6 models (2 facts + 4 dims) with validators; shared across all 3 layers |
| SQL DDL + security | 6-table PostgreSQL schema with FK constraints, 5 indexes, and idempotent `infoagent_readonly` user |
| ShadowTraffic configs | 3 JSON configs generating 160K rows (dims + fato_vendas + fato_estoque) with seasonal patterns |
| Postgres loader + seed | Post-load Pydantic validation with explicit PK map and tabular validation report |
| Knowledge base | 3 Markdown docs indexed into Qdrant via LlamaIndex + HuggingFace BGE embeddings |
| MCP server | Stdio-transport MCP server with 3 tools, SHA-256 query log sanitisation, 500-row cap |
| LangGraph routing graph | 3-node conditional dispatch (classify → type1/type2/type3 → END) |
| Classifier node | Claude call with JSON-only output, KPI name extraction, Qdrant pre-fetch for type2 |
| SQL Agent node | 10-iteration tool_use loop, direct psycopg2 with `json.dumps(default=str)` serialisation |
| RAG Agent node | Qdrant similarity retrieval + Claude synthesis using rag_agent.txt prompt |
| Hybrid Agent node | Sequential SQL + RAG composition → executive Insight → Evidence → Recommendation |
| Chainlit UI | `astream_events` CoT streaming, `on_chain_start`/`on_chain_end` event handling, answered guard |
| Docker Compose stack | 5 services with ordered seeding (`service_completed_successfully`) and health checks |
| Test suite | 7 test files, fully mocked, covering all 10 acceptance tests |

### Architecture

```
Executive Question → Chainlit UI → LangGraph
  ├── classify_query (Claude JSON call)
  │     └── type2: pre-fetch KPI context from Qdrant
  ├── type1_sql → sql_agent_node (tool_use loop → psycopg2 readonly)
  ├── type2_kpi_sql → sql_agent_node (same + kpi_context injected)
  └── type3_hybrid → hybrid_agent_node (sql_agent + rag_agent → synthesis)
```

## Lessons Learned

### Process

- **5-phase AgentSpec workflow scales well for a 52-file project.** Breaking the build into clearly-scoped waves (foundation → data → context → agent → tests) prevented agent collisions and allowed meaningful parallelism.
- **Parallel background agents double throughput when tasks are truly independent.** Waves 2+3 ran 4 agents simultaneously (ShadowTraffic, loaders, MCP server, Qdrant ingest). Each wave completed in the time of the slowest agent rather than the sum of all.
- **Specialist agents produce better output than generalist.** `@shadowtraffic-specialist` knew about `"table"` vs `"topic"` lookup semantics for PostgreSQL connections without being told. `@dw-specialist` correctly identified FATO_ESTOQUE as a semi-additive fact without prompting.
- **Rate limits are a real build risk for long agent chains.** The ShadowTraffic agent hit a rate limit mid-execution during its first run. Having a per-wave plan (rather than one long sequential chain) meant we could restart just that wave, not the whole build.

### Technical

- **`json.dumps(default=str)` over `str().replace("'", '"')` for SQL result serialisation.** The string-coercion approach breaks on `Decimal`, `datetime`, `None`, and apostrophes in string values. `default=str` is the correct escape hatch — caught post-build and fixed before tests were written.
- **ShadowTraffic `"table"` vs `"topic"` for cross-generator lookups.** When using `"kind": "postgres"`, the `lookup` generator must use `"table"` (not `"topic"`). The KB captured this; the specialist applied it correctly.
- **MCP stdio transport doesn't cross Docker network boundaries.** The DESIGN specified `mcp.server_url: http://mcp-server:8080` (HTTP) but the server was built with stdio transport. Resolution: the SQL agent connects directly to PostgreSQL via the readonly user rather than routing through the MCP server. The MCP server is built and available for future external MCP clients.
- **HuggingFace BGE-small embeddings need `Settings.llm = None`** to prevent LlamaIndex from attempting an OpenAI LLM call during index construction. Not documented in the LlamaIndex quickstart; discovered and fixed in the Qdrant ingest agent.
- **`psycopg2.autocommit = True` on the readonly connection.** Without this, each SELECT opens an implicit transaction that is never committed, eventually exhausting connection slots under load.
- **Credentials need to be consistent across configs.** ShadowTraffic JSON hardcodes credentials; docker-compose uses env var defaults. These diverged (ShadowTraffic used `postgres`/`postgres`; compose defaulted to `infoagent_admin`/`changeme`). Fixed with a post-build `sed` pass — future builds should define credentials in `.env` and have generators read them from environment variables where supported.

### Communication

- **Resolving the 3-type taxonomy early unlocked the architecture.** The initial brainstorm revealed that a binary SQL-vs-RAG router was insufficient for the 13 sample questions. Naming Type 2 (KPI-grounded SQL) as a distinct type before design made the LangGraph choice obvious.
- **Stating KPI names explicitly in the classifier prompt is critical.** The classifier.txt prompt lists all 10 KPI names as classification signals. Without this list, the model would classify "What is the gross profit margin?" as type1_sql (it looks like an aggregation) rather than type2_kpi_sql.
- **Assumption A-001 (ShadowTraffic → PostgreSQL)** was identified as critical in DEFINE and resolved in DESIGN. The KB confirming `"kind": "postgres"` support meant no architecture pivot was needed.

### Tools & Libraries

- **LangGraph `StateGraph`** — The `add_conditional_edges` with a `route_query` function that returns a string key is the cleanest way to express 3-way routing. State carries `query_type`, `kpi_context`, `sql_result`, and `final_answer` cleanly across nodes.
- **`testcontainers`** — Useful for integration tests but not required for unit tests. The test generator made the right call to use `unittest.mock` for all unit tests, keeping them fast and dependency-free.
- **Chainlit `astream_events` v2** — The `version="v2"` parameter on `astream_events` is required for LangGraph compatibility. The `on_chain_start`/`on_chain_end` events fire for each node, enabling real-time CoT display without custom streaming code.
- **`python3 -m py_compile`** — The right pre-Docker syntax gate. Catches `SyntaxError` and `IndentationError` across all files even without installing the project's dependencies.
