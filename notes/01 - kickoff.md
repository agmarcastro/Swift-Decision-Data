Generate a synthetic dataset utilizing shadow traffic to simulate the company's database, adhering to the context provided below.

# Objetivo 2

InfoAgent is a team of autonomous agents built upon synthetic retail data. It answers business inquiries by routing them to the appropriate data repository: SQL for precise figures, and vectors for customer sentiment analysis.

### INGEST (Infrastructure and Validation Pipeline)
In this stage, the objective is to ensure that the generated synthetic data flows into the Data Warehouse in a structured, reproducible, and strictly typed manner, laying the groundwork for AI consumption.

Containerization with Docker: Package PostgreSQL and the ShadowTraffic scripts within Docker containers. This creates an isolated development environment and ensures the simulation of the company's database can be spun up in minutes, maintaining consistency across environments.

Strict Validation with Pydantic: Create Pydantic models that perfectly mirror the Star Schema (e.g., ModeloFatoVendas, ModeloDimProduto). These models will act as a bidirectional protection layer: validating the integrity of the data generated by ShadowTraffic prior to insertion, and ensuring that the structured responses returned by the LLM adhere to the expected format (Structured Outputs).


### CONTEXTUALIZE (Semantic Layer and Connectivity)
The database alone is merely storage; it requires business context to prevent the LLM from producing technical hallucinations (e.g., confusing "gross profit" with "total revenue").

Metadata Repository (Qdrant + LlamaIndex): Build a RAG (Retrieval-Augmented Generation) pipeline focused on metadata and business rules, rather than just raw data. Ingest the data dictionary, KPI calculation rules (e.g., how the company calculates VALOR_DESCONTO), and the store organizational chart into Qdrant. LlamaIndex will be used to retrieve this semantic context and inject it into the LLM's prompt before it attempts to write any SQL.

Execution Bridge (PostgreSQL + MCP): Implement the Model Context Protocol (MCP). This is the centerpiece of the architecture. You will create an MCP Server that securely exposes PostgreSQL as a set of standardized tools (e.g., list_tables(), describe_schema(FATO_VENDAS), execute_read_only_query(sql)). This allows the LLM to dynamically interact with the database, inspecting the structure before extracting the answers.

### AGENT (Orchestration and C-Level Interface)
The final layer transforms isolated resources into a team of autonomous experts accessible through a natural language interface.

Routing and Orchestration (LangChain / CrewAI): Design the system's decision-making flow. Implement a Router Agent that receives the executive's question and delegates it to the appropriate sub-agent:

Data Analyst Agent (SQL): Uses the MCP server to convert the inquiry into a highly precise PostgreSQL query and return the exact figures.

Business Rules Agent (RAG): Consults Qdrant to explain how a metric is composed or to retrieve departmental context.

Role Definition (AgentSpec): Configure detailed System Prompts and guardrails for each agent. This ensures that the SQL Agent, for instance, has strict READ-ONLY access and is explicitly instructed to always optimize queries for low latency.

Executive User Experience (Chainlit): Develop the chat interface. Chainlit will not only receive questions but also display the agents' Chain of Thought in real-time. Executives will be able to watch the agent fetch the table schema, write the SQL, validate the data, and ultimately plot the results in tables or charts directly within the chat interface.
